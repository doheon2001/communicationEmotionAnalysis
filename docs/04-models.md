# 4. 분류 모델

## 분류가 뭔가요

**분류(classification)**는 입력을 정해진 칸 중 하나로 나누는 것입니다.

- 감정 분류: 문장 → `긍정` 또는 `부정`
- 공격성 탐지: 문장 → `공격적` 또는 `비공격적`

ChaeOn의 ML 모델 두 개(기능 1·2)는 모두 분류 모델입니다.
입력은 [03 문서](03-vectorization.md)에서 만든 **벡터(숫자 목록)**입니다.

기획서가 후보로 든 모델은 세 가지입니다. 하나씩 비유로 보겠습니다.

## 로지스틱 회귀 (Logistic Regression)

**아이디어**: 각 단어에 점수를 매겨 다 더한 뒤, 그 합을 확률로 바꿉니다.

> 단어 저울로 생각하세요. `좋았어` = +3점, `별로` = -4점, `짜증` = -5점 …
> 문장에 든 단어 점수를 모두 더해서 합이 양수 쪽이면 긍정, 음수 쪽이면 부정.
> 그리고 그 합을 0~1 사이 **확률**로 눌러 줍니다.

- 장점: 단순하고 빠름. **각 단어가 결과에 얼마나 기여했는지 보임** (해석 쉬움).
- 텍스트 분류의 **기본 베이스라인**으로 가장 많이 쓰임.
- ChaeOn: 기능 1·2 **양쪽 모두**의 주력 후보.

## 나이브 베이즈 (Naive Bayes)

**아이디어**: 확률 계산. "이 단어들이 나왔을 때, 부정일 확률은?"

> 고전적인 **스팸 메일 필터**가 이 방식입니다. "무료", "당첨" 같은 단어가
> 스팸에서 자주 나왔다는 걸 확률로 기억해 두고, 새 메일에 그 단어들이 보이면
> 스팸 확률을 높게 계산합니다.
> "Naive(순진한)"인 이유: 단어들이 서로 독립이라고 (단순하게) 가정해서요.

- 장점: 매우 빠르고, 데이터가 적어도 그럭저럭 작동.
- 텍스트 분류에서 의외로 강한 편.
- ChaeOn: 기능 2(공격성 탐지) 후보.

## 랜덤 포레스트 (Random Forest)

**아이디어**: **결정 트리(스무고개)** 여러 개를 만들어 다수결 투표.

> 결정 트리 하나는 스무고개입니다 — "`짜증` 단어 있나? → 예 → `진짜` 있나? …".
> 트리 하나만 쓰면 편향될 수 있어서, 조금씩 다른 트리를 **수십~수백 개** 만들고
> 그 결과를 **투표**로 합칩니다. "여러 명에게 물어보고 다수결" = 숲(forest).

- 장점: 복잡한 패턴도 잘 잡고, 한 트리의 실수를 다른 트리가 보완.
- 단점: 로지스틱 회귀보다 무겁고 해석이 덜 직관적.
- ChaeOn: 기능 2(공격성 탐지) 후보.

## 어떤 기능에 어떤 모델

| 기능 | 후보 모델 (기획서) |
| --- | --- |
| 기능 1 — 감정 분류 | 로지스틱 회귀 (+ TF-IDF / CountVectorizer) |
| 기능 2 — 공격성 탐지 | 로지스틱 회귀 / 랜덤 포레스트 / 나이브 베이즈 |

여러 후보가 있는 이유: **미리 정답이 없습니다.** 다 학습시켜 보고
[05 문서](05-evaluation.md)의 지표로 비교해서 제일 나은 걸 고릅니다.

## 중요: predict vs predict_proba

분류 모델은 결과를 두 가지 형태로 줍니다.

| 메서드 | 출력 | 예시 |
| --- | --- | --- |
| `predict` | 칸 이름 (딱 잘라서) | `"부정"` |
| `predict_proba` | 각 칸의 **확률** | `긍정 0.13, 부정 0.87` |

ChaeOn 기획서의 출력 형태를 다시 보세요:

- 기능 1: "긍정 0.87, 부정 0.13" ← **`predict_proba`의 결과**
- 기능 2: "공격성 True/False" ← `predict`, "공격성 확률 0.82" ← `predict_proba`

즉 ChaeOn은 단순 라벨이 아니라 **확률값**이 필요하므로 `predict_proba`를
주로 씁니다. 확률이어야 기능 3·5에서 "부정 비율이 평소 대비 몇 배" 같은
**정도(degree)** 분석이 가능하기 때문입니다.

> 참고: K-HATERS 데이터가 공격성을 3점 척도로 제공하는 것도, 단순 True/False가
> 아니라 **강도가 있는 확률 출력**을 만들기 좋기 때문입니다.

## 학습한 모델은 어디에 저장하나

학습이 끝난 모델은 파일로 저장해 둡니다 (예: `data/` 바깥의 `models/` 폴더).
저장해야 매번 다시 학습하지 않고 **바로 예측에 쓸 수 있습니다.**
이때 [03 문서](03-vectorization.md)의 vectorizer도 **같이** 저장하세요 —
모델과 벡터화 규칙은 항상 한 쌍으로 움직입니다.
