# 3. 텍스트 벡터화

## 왜 필요한가

모델은 **글자를 못 읽습니다.** 숫자만 계산할 수 있습니다.
그래서 "회의 진짜 좋았어" 같은 문장을 `[1, 0, 2, 1, ...]` 같은 **숫자 목록**으로
바꿔 줘야 합니다. 이 변환을 **벡터화(vectorization)**라고 하고, 결과 숫자 목록을
**벡터(vector)**라고 부릅니다.

벡터화 전에 보통 **전처리**를 먼저 합니다.

## 전처리 — 벡터화 전 청소

채팅 텍스트는 지저분합니다. 그대로 벡터화하면 "좋아", "좋아!!", "좋아ㅋㅋ"가
전부 다른 단어로 취급됩니다. 그래서 먼저 다듬습니다.

| 작업 | 예시 |
| --- | --- |
| 특수문자·이모지 제거 | `좋아요!! 😄` → `좋아요` |
| 중복 공백 정리 | `회의   진짜` → `회의 진짜` |
| 토큰화 (단어로 쪼개기) | `회의진짜좋았어` → `회의 / 진짜 / 좋았어` |

> **토큰화**: 문장을 의미 단위(토큰)로 쪼개는 것. 한국어는 띄어쓰기만으로는
> 부족해서 `KoNLPy` 같은 형태소 분석기를 자주 씁니다.
> ChaeOn에서: 채팅 특유의 "ㅋㅋ", 자음 표현, 오타 처리 방침을 팀이 정해야 합니다.

청소된 문장은 `data/processed/`에 저장됩니다 → [02 문서](02-data-structure.md).

## 방법 1 — CountVectorizer (단어 세기)

가장 단순한 방법. **"각 단어가 문장에 몇 번 나왔나"**를 셉니다.
이 방식을 **BoW(Bag of Words, 단어 가방)**라고 부릅니다 — 단어를 가방에
쏟아 넣고 개수만 세는 느낌.

문장 두 개로 예를 들어 봅시다.

- 문장 A: `회의 진짜 좋았어`
- 문장 B: `회의 진짜 별로였어`

먼저 전체에 등장한 단어로 **사전**을 만듭니다:
`[회의, 진짜, 좋았어, 별로였어]`

이제 각 문장을 "사전 순서대로 개수"로 적습니다:

| | 회의 | 진짜 | 좋았어 | 별로였어 |
| --- | --- | --- | --- | --- |
| 문장 A | 1 | 1 | 1 | 0 |
| 문장 B | 1 | 1 | 0 | 1 |

→ 문장 A = `[1, 1, 1, 0]`, 문장 B = `[1, 1, 0, 1]`. 이게 벡터입니다.

## 방법 2 — TF-IDF (중요한 단어에 가중치)

CountVectorizer의 약점: 위 예에서 `회의`와 `진짜`는 두 문장 모두에 있어서
**감정을 구별하는 데 전혀 도움이 안 됩니다.** 그런데 개수는 똑같이 1로 셉니다.

**TF-IDF**는 여기에 가중치를 줍니다. 핵심 아이디어:

> "모든 문장에 흔하게 나오는 단어는 덜 중요하다.
>  특정 문장에만 나오는 단어가 더 중요하다."

- **TF** (Term Frequency): 이 문장에 단어가 자주 나올수록 ↑
- **IDF** (Inverse Document Frequency): 그 단어가 **다른 문장들에도 흔하면** ↓

위 예에 적용하면:

| | 회의 | 진짜 | 좋았어 | 별로였어 |
| --- | --- | --- | --- | --- |
| 가중치 경향 | 낮음 (흔함) | 낮음 (흔함) | **높음** (A에만) | **높음** (B에만) |

→ `좋았어`/`별로였어`처럼 **감정을 가르는 단어가 큰 숫자**를 받습니다.
모델이 감정·공격성을 구별하기 훨씬 쉬워집니다.

## CountVectorizer vs TF-IDF

| | CountVectorizer | TF-IDF |
| --- | --- | --- |
| 하는 일 | 단어 개수만 셈 | 개수 + 흔함/희귀함 가중치 |
| 흔한 단어 | 그대로 셈 | 자동으로 약하게 |
| 보통 성능 | 기본 | 더 좋은 경우 많음 |
| ChaeOn | 비교용 베이스라인 | 주력 후보 |

## ChaeOn에서

기획서상 ChaeOn은 **CountVectorizer와 TF-IDF를 둘 다** 후보로 둡니다.
실제로는 둘 다 만들어 보고 [05 문서](05-evaluation.md)의 지표로 비교해서
**더 점수가 높은 쪽을 채택**하면 됩니다.

만든 벡터와 **변환 규칙(vectorizer 객체)**은 `data/features/`에 저장합니다.
규칙을 저장해야 하는 이유: 실제 채팅을 예측할 때 **학습 때와 똑같은 규칙**으로
벡터화해야 하기 때문입니다 → [01 문서](01-ml-pipeline.md) 마지막 절 참고.
